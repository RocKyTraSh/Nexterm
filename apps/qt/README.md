# Qt frontend (apps/qt)

Статус: заготовка. GUI появляется в v0.2. Сейчас бинарь только печатает
сообщение, чтобы workspace собирался целиком.

## Почему GUI не в первом MVP
Терминальный GUI с вкладками, split-панелями и рендерингом VT — самая
трудоёмкая часть всего проекта. Сначала фиксируется UI-agnostic ядро
(`rrs-ui-common::AppCore`), затем поверх него строится фронтенд. Так и Qt, и
GTK переиспользуют одну и ту же логику, а не дублируют её.

## Выбранный подход (приоритетный)
**Rust core + Qt/QML через `cxx-qt` (KDAB):**
- вся бизнес-логика и протоколы — в Rust-крейтах;
- QML описывает окно, вкладки, sidebar и диалоги;
- `cxx-qt` экспонирует Rust-объекты (модели сессий, SFTP, туннелей) как
  `QObject` со свойствами/сигналами/слотами для QML.

Почему так:
- нативный вид в KDE/Plasma и поддержка тёмной/светлой темы средствами Qt;
- `cxx-qt` — наиболее поддерживаемый сегодня способ связать Rust и Qt;
- сборка через CMake/qmake уживается с cargo.

## Рассмотренные альтернативы
1. **Тонкий C++/Qt Widgets фронтенд + Rust core через `cxx` (FFI).**
   Плюс: можно взять готовый терминальный виджет QTermWidget (на базе Konsole).
   Минус: больше C++-кода и ручного FFI.
2. **Отдельный Rust-демон + Qt-клиент по IPC (JSON-RPC поверх unix-сокета).**
   Плюс: жёсткая изоляция UI и движка, удобно для headless-режима.
   Минус: больше движущихся частей, сложнее отладка.
3. **Чистый Rust GUI (egui / iced / slint).**
   Не выбрано как приоритет: хуже нативная интеграция с KDE, слабее готовые
   терминальные виджеты.

## Самая сложная часть — терминальный виджет
В QML нет готового VT-терминала. Варианты:
- рисовать сетку ячеек самостоятельно поверх модели из `alacritty_terminal` или
  `vte` (парсер VT) — максимально гибко, но трудоёмко;
- использовать C++ QTermWidget (тогда логично выбрать вариант 1) и прокидывать
  байты между виджетом и SSH-бэкендом.
Решение фиксируется в начале v0.2.
